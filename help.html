Help

A practical beta workflow, kept intentionally simple.

Legacyless is designed to fit into a normal developer workflow: point it at a real project, preview a focused rule, inspect the output, then apply only when you are comfortable with the change.

Beta access

What you need first

  • A Legacyless beta binary or release package
  • A local checkout of the C++ project you want to inspect
  • A valid compile_commands.json compilation database

Requirements

Legacyless works best when build context is available.

The tool needs enough information to reason about your code as it is actually compiled. In practice, that means using it with a project checkout and a compilation database rather than isolated files.

Project source tree

Run the beta against the real codebase you want to modernize, not a disconnected copy-paste sample.

Compilation database

Legacyless expects compile_commands.json so it can understand your project with the same compilation context your tooling already uses.

Review-friendly workflow

Keep Git available so you can compare previews, inspect diffs, and adopt changes incrementally.

Typical flow

Start small and inspect everything.

01

Install or unpack the beta

Download the current beta artifact and place the binary somewhere convenient for local testing.

02

Generate compile_commands.json

If you use CMake, enable exported compile commands. If you use another build system, generate an equivalent compilation database.

03

Run preview first

Target one rule, inspect the proposed diff, and make sure the output matches the actual code intent.

04

Use apply when ready

Apply only after the preview looks clean, and keep the resulting commit focused so review stays easy.

Example

Illustrative command flow

# Example compilation database with CMake
cmake -S . -B build -DCMAKE_EXPORT_COMPILE_COMMANDS=ON

# Preview a specific rule
legacyless preview --rule <rule-name> --compdb build/compile_commands.json

# Apply once the preview looks correct
legacyless apply --rule <rule-name> --compdb build/compile_commands.json

The exact beta packaging and flags may evolve. Treat the commands above as a practical model you can align with the current release notes.

Troubleshooting

Short notes for the most common early beta issues.

No compile_commands.json

Generate it from your build system first. Without a compilation database, the tool may not have enough project context to analyze the code safely.

Unexpected preview output

Start with a smaller scope, run a single rule at a time, and compare the result against the code's real intent before applying anything broadly.

Beta feedback matters

If something looks wrong, unclear, or incomplete, send the sample and the rule name. That feedback is exactly what helps the product become trustworthy.

Need a contact point?

Send beta feedback with real examples.

The most useful reports include the rule you ran, what you expected, what Legacyless produced, and enough surrounding context to reproduce the case.
