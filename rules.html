Rules

Small, reviewable rules with clear intent.

The current Legacyless ruleset is intentionally narrow. Each rule is meant to be practical, conservative, and easy for developers to understand before it earns a place in an automated workflow.

Current catalog

Two foundational rules

  • nullptr modernization for legacy null pointer constants
  • Missing override annotation where override semantics already exist

This tight scope is deliberate. Legacyless grows rule-by-rule, with testing and feedback guiding what should be automated next.

Rule 01 nullptr modernization

Replace legacy null pointer constants with nullptr

Human-friendly description: when code is expressing a null pointer, Legacyless can rewrite the older spelling to the modern, explicit form.

Before

Connection* current = NULL;
if (current == 0) {
  reconnect(current);
}

After

Connection* current = nullptr;
if (current == nullptr) {
  reconnect(current);
}

Short explanation

nullptr communicates pointer intent directly and avoids the historical ambiguity of NULL and plain 0.

Why it is safe

The rule is only appropriate when Legacyless can determine the token is being used as a null pointer constant in pointer context. It is not about broad replacement of every zero literal.

Why developers like it

The change is small, visible in a diff, and aligns with modern C++ style without changing the surrounding design or control flow.

Rule 02 missing override

Add missing override where override behavior already exists

Human-friendly description: if a derived member function is already overriding a base virtual function, Legacyless can mark that intent explicitly.

Before

struct Writer : BaseWriter {
  void flush();
};

After

struct Writer : BaseWriter {
  void flush() override;
};

Short explanation

The rule adds the missing annotation for a function that is already acting as an override according to the type system.

Why it is safe

Adding override does not create new runtime behavior. It makes existing intent explicit and lets the compiler diagnose mismatches earlier if the hierarchy changes later.

Why developers like it

Reviewers can understand the change instantly, and the code becomes clearer about which member functions are meant to participate in polymorphism.
